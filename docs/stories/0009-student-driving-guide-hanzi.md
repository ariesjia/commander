# 学生端「驾驶指南」：一年级一类字词语手写练习

> 概述：在学生端「机甲维修」入口**下方**增加**「驾驶指南」**入口（叙事上可解释为「出舱前识字校准 / 控制台词条确认」等）。首期面向**小学一年级一类字**相关**词语**：**每次会话固定练 10 个汉字**（按词语拆成多步，例如若干两字词合计十字）；**每个字都展示带声调拼音**，学生在**手写区书写完整词语**（一步一词、提交判定）。**不使用语音朗读**（无题目 TTS、无反馈 TTS）。**手写判定成功**时给予**鼓励**（文案/动效）；**失败**时给予**提示**并可重试。**关于手写识别**：纯浏览器内通用手写识别难度高，本 story 仍保留 §3.1 分期方案（弱验证 / 云端识字 API 等）。

**关联**：[`0007-student-daily-maintenance-math`](./0007-student-daily-maintenance-math.md)（每日一次小游戏、家长开关、无积分等模式可对齐）；学生主页 [`src/app/student/page.tsx`](../../src/app/student/page.tsx)；`Parent` 设置与家长开关模式；课标一类字词表需产品/教研提供或引用公开词表种子。

---

## 1. 用户故事

**作为** 学生  
**我想要** 每次用大约十字的练习，看着**每个字上面的拼音**，在屏幕上手写出**整个词语**；写对时有鼓励，写不对时有提示让我再试  
**以便** 在 MotiMech 里练一年级一类字词语，又不用听语音、可以安静专注写

**作为家长 / 产品**  
**我想要** 能**开启或关闭**「驾驶指南」（默认与维修类似可配置），并理解**手写判分**依赖技术方案（本地笔画 / 云端识字 / 仅自评）的差异  
**以便** 按家庭对屏幕时间、隐私与准确性的要求做选择

## 2. 验收标准（Acceptance Criteria）

### 2.0 练习结构（已定）

| 项目 | 说明 |
|------|------|
| 每次会话字数 | **10 个汉字**（例如 5 个两字词，或 3+3+4 等组合，由出题器按词表与种子生成，**合计十字**） |
| 单步任务 | 呈现**一个词语**；该词中**每个字单独展示带声调拼音**（如「中 zhōng」「国 guó」排版实现时统一） |
| 手写目标 | 学生在手写区书写**整个词语**（非单字拆多次提交，除非产品改为逐字——本文以**整词一次提交**为准） |
| 语音 | **不需要**：不朗读题目、不朗读反馈（**禁用** `speechSynthesis` / TTS 于本功能） |
| 成功 | **鼓励**：至少一种明确反馈（如动画 + 文案池随机一句，风格与机甲维修一致、积极） |
| 失败 | **提示**：说明可再试、笔画或识别小贴士（避免羞辱性用语）；允许清除重写、再次提交 |

### 2.1 入口与范围

- [ ] 学生主页在「机甲维修」区块**下方**展示「驾驶指南」入口（维修关闭时仍可单独展示驾驶指南，除非产品规定捆绑展示——建议**独立开关**）。
- [ ] 每日频次、自然日边界与 `0007`/`0002` 一致：**Asia/Shanghai**，**每自然日可完成次数**由产品定（建议首期 **1 次/日**，与维修对齐便于心智模型）。
- [ ] 首期词库：**一年级一类字构成的词语**，**全部先配置在代码库内的词库文件**（见 **§2.4**）；实现层抽象「词目 + 逐字拼音」；会话从词库**抽样**若干词拼满 **10 字**（候选池字数可大于 10）。

### 2.2 交互与反馈

- [ ] 分步展示：每一步仅当前**词语**；**词语内每个字均显示拼音**（带声调）；不在本功能内使用语音朗读。
- [ ] 提供**手写区域**（Canvas 或覆盖层），支持触控与触控笔；有「清除」「提交」等基础操作；提交后对**整词**做识别比对（技术见 §3.1）。
- [ ] **成功**：展示鼓励类反馈（文案常量池 + 轻量动效，可复用维修成功情绪等级、换皮驾驶舱叙事）。
- [ ] **失败**：展示提示类反馈（引导重试、简短原因如「再写工整一点」或识别未命中目标词）；文案避免「不及格」「笨」等；支持当场重试直至该步通过或达最大次数（次数策略可配置）。
- [ ] 全部 **10 字**对应的各步完成后，进入**完成页**（与维修「今日完成」类似）。

### 2.3 家长侧

- [ ] `Parent` 增加开关字段（如 `drivingGuideEnabled`），默认 `true`；关闭后学生端隐藏入口且相关 API 返回业务错误码。
- [ ] 家长端设置页可开关（与机甲维修、注音等并列），说明文案需一句解释「识字练习，不产生积分」。

### 2.4 词库配置文件与首期词条

**配置文件（建议路径，实现时可微调）**

- 词库以 **TypeScript 或 JSON 配置文件**维护，与 [`src/config/maintenance-math.ts`](../../src/config/maintenance-math.ts) 类似，例如：

## 3. 技术方案

### 3.1 手写识别与打分：**是否可能？**

| 方向 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **A. 纯浏览器 + 深度学习模型** | TensorFlow.js / ONNX Runtime Web 等加载中文字写模型 | 无学生笔迹出网 | 包体大、端上性能与准确率波动、维护成本高 |
| **B. 服务端 + 第三方 OCR / 手写识字 API** | 提交笔画栅格图或快照由云端返回识别文本 | 准确率通常优于轻量端侧模型 | 费用、延迟、**隐私与合规**（未成年人笔迹图像）、需密钥管理 |
| **C. 笔画/笔顺教学型（如 Hanzi Writer 思路）** | 对比标准字笔画顺序与覆盖区域，偏「写对笔顺」而非草书识别 | 端上可算「分数」、无出网 | 与「自由手写」体验不一致，需产品接受 |
| **D. 弱验证 MVP** | 不识别字形：仅记录完成时长 + 家长/学生自评，或「写完后揭晓答案」对照 | 实现快、零合规风险 | **无自动判对错** |

**结论**：**自动识别与打分在工程上可行**，但「像老师一样给手写质量打分」需要么选用 **B**，要么用 **C** 换问题定义（笔顺/描红）；**纯前端任意手写实时识别**不推荐作为首期唯一方案。

建议在 story 内固定**分期**：

1. **Phase 1（可上线）**：手写区 + 提交快照 + **服务端调用识字 API**（或先做 **D** 积累数据）→ 与目标字比对，**通过/不通过**（可选置信度阈值）。
2. **Phase 2**：优化词库、提示策略、离线缓存、错误时的引导动画；若不上云，再评估 **C** 作为低龄模式。

#### Tesseract 能否实现？

- **技术上能集成**：Tesseract 4/5 支持 **简体中文**（如 `chi_sim`），可在服务端对画布导出的 **PNG/JPEG** 跑 OCR，再与目标词语做字符串比对。
- **不适合作为本场景的主识别方案**：Tesseract 主要面向**印刷体**与相对规整的扫描件；**小学手写汉字**（笔画夸张、连笔、出格、指绘粗细不匀）识别率通常**偏低**，误判多，难以稳定支撑「写对即通过」的产品体验。
- **若坚持用开源、不上商业手写 API**：可把它当作**原型/实验**（工整大字、白底、字占画面比例高），或改产品形态为「展示印刷词 → OCR 辅助」而非「自由手写判题」。更贴近需求的开源替代往往是**笔顺/描红类**（见上表 **C**），而非通用 OCR。

### 3.4 前端设计（Frontend Design）

- **页面**：`/student/driving-guide`（与 `/student/maintenance-math` 并列）；`StudentPageHeader` 返回 `/student`。
- **组件**：手写区推荐封装 `HandwritingCanvas`（`touch`/`pointer` 事件、高清屏 `devicePixelRatio`、清除/撤销可选）。
- **交互**：「准备页 → 开始 → **多步词语（每步合计会话内进度，共 10 字）** → 完成页」；每步展示**逐字拼音** + 整词手写提交；若 API 异步识别，需 loading 与失败重试。
- **文案常量**：单独文件维护 **SUCCESS_ENCOURAGEMENT_LINES**、**FAILURE_HINT_LINES**（或按失败原因分桶），与 [`MAINTENANCE_COPY`](../../src/lib/maintenance-math/copy.ts) 模式类似，便于调语气与 A/B。
- **无障碍**：本 story 不要求 TTS；如需读屏，可后续加 `aria-label`（非语音自动播）。
- **视觉**：学生端科技卡通、与维修区块区分色系（例如靛蓝/紫系避免与 teal 维修混淆）；主按钮 ≥44px。

### 3.5 识别服务（实现时选型）

- **MVP 测试（已定）**：使用 **Tesseract** 在手写截图上做 OCR，与目标词做字符串比对；实现细节、CLI vs `tesseract.js`、预处理与局限见仓库 **[docs/plans/driving-guide-hanzi-mvp.md](../plans/driving-guide-hanzi-mvp.md)**。对手写少儿字误判率可能较高，本阶段作**基线实验**，后续可换云手写 API 等而不改产品主流程。
- 若采用云端：在**服务端**持有密钥，前端只传**临时签名 URL 或 base64**（注意体积）；日志中**避免长期存原图**（可仅存是否通过 + 词 id）。
- 若采用端侧笔画库：明确产品文案为「笔顺练习」而非「任意潦草字识别」。

## 4. 依赖与风险

- **依赖**：拼音展示与现有 `showPinyin`、任务列表注音可复用经验；词表需教研或版权清晰来源。
- **风险**：第三方 API 成本与稳定性；未成年人图像数据需最小化采集与隐私条款；纯端识别准确率期望管理。
